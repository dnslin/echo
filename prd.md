## 2. 产品定位

### 2.1 一句话定位

> echo 是一款给朋友开黑用的轻量多人语音工具，打开即用，输入邀请码即可进入临时语音房间。

### 2.2 背景问题

一部分 PC 游戏玩家只需要“进房间、能说话、能听见”，但常见语音软件容易出现以下问题：

| 痛点 | 表现 | echo 的应对 |
| --- | --- | --- |
| 软件太重 | 启动慢、界面复杂、占用感强 | 保持功能克制，优先完成语音核心链路 |
| 进房间麻烦 | 注册、加好友、找频道、建服务器 | 昵称 + 邀请码即可加入临时房间 |
| 游戏中不方便操作 | 切窗口麻烦，按键说话不稳定 | 支持系统托盘和后台按键说话 |
| 状态不清 | 不知道自己是否正在发声、是否断线 | 明确展示语音、静音、连接和输入反馈 |
| 不想被陌生人打扰 | 社区、频道、公开入口带来噪音 | MVP 只支持朋友私下分享邀请码 |

### 2.3 核心价值

```text
轻量
快速进房
低打扰
按键说话稳定
多人语音清晰
适合 Windows 游戏场景
```

### 2.4 产品不是

```text
不是 Discord 平替
不是大型社区平台
不是直播工具
不是视频会议软件
不是社交软件
不是聊天室
不是公会 / 战队系统
```

## 3. 目标用户与成功定义

### 3.1 首批目标用户

| 用户类型 | 描述 | 核心诉求 |
| --- | --- | --- |
| PC 游戏玩家 | 经常和朋友一起玩电脑游戏 | 快速语音、低占用、少打扰 |
| 熟人开黑小队 | 2-10 人左右，经常一起玩 | 快速创建临时房间，方便进出 |
| 临时组队玩家 | 临时拉朋友玩一局 | 不想注册复杂账号，只想输入邀请码 |
| 对现有语音软件不满意的人 | 觉得现有软件臃肿、复杂 | 一个干净的小工具 |

第一阶段重点服务：

```text
Windows 10 / Windows 11 x64 用户
2-10 人小队
朋友之间私下使用
游戏期间语音沟通
接受产品处于朋友内测状态
```

### 3.2 MVP 目标

MVP 只追求一个目标：

> 让一群朋友在 Windows 电脑上，可以快速进入同一个临时房间，并稳定完成游戏语音交流。

### 3.3 MVP 必达成功条件

```text
1. 首次用户可以设置昵称和随机头像。
2. 用户可以快速创建临时房间并获得邀请码。
3. 用户可以通过 6 位邀请码加入临时房间。
4. 两名用户可以进入同一房间并双向语音。
5. 3-10 名用户可以在同一房间进行语音交流。
6. 第 11 名用户无法加入，并看到明确提示。
7. 用户可以按住指定快捷键说话，松开后停止说话。
8. 应用在后台、游戏在前台时，按键说话仍应可用。
9. 用户可以主动切换为自由说话，但不会默认自动开麦。
10. 用户可以静音和取消静音自己。
11. 用户可以看到房间成员、谁正在说话、谁静音、谁重连中。
12. 用户可以选择麦克风设备和输出设备。
13. 用户可以测试麦克风输入，并调节整体房间语音音量。
14. 用户关闭窗口后，应用进入系统托盘，语音不中断。
15. 网络异常时有重连状态、失败提示和可操作按钮。
16. Windows 10 / Windows 11 x64 下可以正常安装、启动和使用。
```

## 5. 核心概念与边界

### 5.1 临时房间

临时房间是一次朋友开黑会话使用的语音房间。用户通过邀请码邀请朋友加入。MVP 不承诺长期保留，也不承诺第二天仍可用。

生命周期：

```text
房间创建成功后，邀请码立即有效。
有成员在线时，房间保持有效。
最后一名成员离开后，房间保留 30 分钟。
30 分钟内有人通过原邀请码加入，则房间恢复有效。
超过 30 分钟无人加入，则房间失效，邀请码不可再用。
如果创建成功后 30 分钟内没有任何成员进入，房间失效。
```

### 5.2 邀请码

```text
邀请码为 6 位大写字母数字短码。
用户输入时大小写不敏感。
用户输入时忽略空格和短横线。
邀请码只用于加入临时房间，不是房间密码。
过期房间的邀请码不可再用。
```

复制邀请内容示例：

```text
加入我的语音房间，邀请码：K7M9Q2
请打开 echo 应用后输入邀请码加入。
```

MVP 不做房间链接、二维码、一键分享到 QQ/微信、长期邀请码或自定义邀请码。

### 5.3 本地匿名身份、昵称和随机头像

| 概念 | MVP 规则 |
| --- | --- |
| 匿名身份 | 应用在当前电脑上生成并保存，用于识别本机用户和支持重连；不是账号 |
| 昵称 | 房间成员列表中的展示名；不唯一；同房间允许重复昵称 |
| 随机头像 | 为本机匿名身份生成并保存；允许重新随机；不支持上传、裁剪或头像库 |

MVP 不做跨设备同步、找回身份、手机号、邮箱、好友关系或云端资料。

### 5.4 房主

房主是创建临时房间的用户。MVP 中房主只是房间内的身份标识。

```text
房主没有踢人权限。
房主不能撤销邀请码。
房主不能转让房主。
房主不能关闭房间。
```

如邀请码泄露或陌生人进入，MVP 的处理方式是：离开当前房间，重新创建临时房间，分享新的邀请码。

### 5.5 本地设置

MVP 保存以下本地设置：

```text
匿名身份
昵称
随机头像
按键说话快捷键
麦克风设备
输出设备
语音模式偏好
整体房间语音音量
```

MVP 不保存：

```text
关闭窗口行为偏好
房间历史
自动加入上次房间
账号资料
云端配置
```

如果保存的语音模式偏好是自由说话，下次进入房间也不能自动发送声音。

## 6. 核心用户流程

### 6.1 首次启动

```text
打开 echo
如果本机没有昵称 → 进入昵称页
用户输入昵称
系统生成随机头像
用户进入首页
```

验收：首次打开必须设置昵称；昵称保存本地；再次启动直接进入首页；同房间允许重复昵称。

### 6.2 创建临时房间

```text
首页点击“创建房间”
可选输入房间名
创建成功后显示房间名、邀请码、复制邀请按钮、进入房间按钮
用户复制邀请码发给朋友
用户点击进入房间
```

验收：用户可以在极少步骤内创建临时房间并获得邀请码；创建成功页清楚展示邀请码；复制后有轻反馈。

### 6.3 加入临时房间

```text
首页输入邀请码
应用归一化邀请码
服务校验房间是否有效、是否过期、是否满员
有效则进入房间页
无效则显示可操作错误
```

加入失败场景：

| 场景 | 文案 | 操作 |
| --- | --- | --- |
| 邀请码为空 | 请输入邀请码 | 输入邀请码 |
| 邀请码格式错误 | 邀请码应为 6 位字母或数字 | 重新输入 |
| 邀请码无效 | 邀请码无效，请检查后重试 | 重新输入 / 返回首页 |
| 房间过期 | 该房间已过期，请让朋友重新创建 | 返回首页 / 创建新房间 |
| 房间人数满 | 房间人数已满，暂时无法加入 | 返回首页 / 重试 |
| 连接失败 | 连接失败，请检查网络后重试 | 重试 / 返回首页 / 反馈问题 |

### 6.4 房间内使用

房间页必须优先展示：

```text
自己的语音状态
连接状态
按键说话状态
成员列表
静音入口
离开房间入口
设置入口
```

房间页不应优先展示设置项、关于信息、版本信息或反馈入口。

### 6.5 离开、关闭窗口和退出应用

| 用户操作 | 产品行为 |
| --- | --- |
| 点击“离开房间” | 轻确认后离开当前房间，停止发送和接收房间语音，返回首页 |
| 点击窗口右上角 X | 不离开房间，不退出应用，进入系统托盘，语音继续保持 |
| 托盘菜单点击“显示主窗口” | 恢复主窗口 |
| 托盘菜单点击“退出应用” | 退出前自动离开当前房间，停止语音，关闭应用 |
| 应用崩溃 / 电脑关机 / 进程被杀 | 视为异常断开，其他成员短时间看到连接异常，超过重连窗口后从成员列表移除 |

## 7. 功能需求

### 7.1 语音房间页与成员列表

房间页内容：

```text
房间名称
邀请码
复制邀请按钮
连接状态
成员列表
自己的语音状态
语音模式切换
自己静音 / 取消静音
当前按键
按键状态
简化输入反馈
设置入口
离开房间按钮
```

成员展示：

```text
随机头像
昵称
是否为“我”
是否为房主
是否正在说话
是否自己静音
是否重连中 / 连接异常
```

排序规则：

```text
“我”的状态固定在最上方或单独的“我的状态”区域。
其他成员按加入房间时间保持稳定顺序。
正在说话的成员通过高亮展示，不因为说话状态跳到列表前面。
房主显示房主标识，但不要求排在第一位。
重连中成员保留在原位置，显示重连中 / 连接异常状态。
```

验收：用户可以看到当前成员、谁正在说话、谁静音、谁重连中、自己的按键状态和输入反馈；成员列表不会因为说话状态频繁跳动。

### 7.2 正在说话与输入反馈

正在说话定义：

```text
正在说话 = 检测到有效语音输入。
不是单纯麦克风发送通道打开。
```

规则：

```text
按住快捷键但没有声音输入时，不一定对其他成员显示“正在说话”。
自由说话开启但环境安静时，不一定对其他成员显示“正在说话”。
其他成员只在检测到有效输入时显示正在说话。
成员停止说话后，说话高亮可短暂保留约 1 秒，避免短句说话者难以识别。
```

自己的输入反馈至少显示：当前是否检测到声音、当前是否正在说话。按键说话模式下还应显示当前快捷键和按键状态。

### 7.3 按键说话

默认行为：

```text
默认语音模式：按键说话
默认快捷键：V
按下快捷键：满足发送条件
松开快捷键：停止发送
首次进入房间默认不开持续麦
```

快捷键设置：

| 类型 | MVP 支持 |
| --- | --- |
| 键盘单键 | 支持 |
| 鼠标侧键 4 / 5 | 支持 |
| 组合键 | 不重点承诺 |
| 鼠标宏键 / 手柄 / 多键复杂组合 / 游戏内虚拟按键 | 不承诺 |

禁止设置：

```text
Esc
Win
Alt+F4
鼠标左键
鼠标右键
Enter
```

后台游戏边界：

```text
当 echo 不在前台，且用户正在全屏、无边框窗口或窗口化游戏中时，按键说话快捷键仍应被识别。
MVP 不承诺所有带反作弊系统的游戏都 100% 可识别。
MVP 不承诺管理员权限游戏下 100% 可识别。
MVP 不承诺远程桌面或虚拟机里的按键一定可用。
```

按键设置区域应提供固定排障说明：

```text
如果快捷键在游戏中无效：
1. 尝试更换快捷键，例如鼠标侧键或 CapsLock。
2. 确认游戏没有占用该按键。
3. 如果游戏以管理员权限运行，请尝试以相同权限启动本应用。
4. 部分带反作弊的游戏可能限制全局快捷键。
```

验收：用户按住快捷键时满足发送条件，松开后停止发送；页面清楚显示当前按键、按键是否被识别；修改后立即生效并保存；后台游戏前台时仍应可用。

### 7.4 自由说话

自由说话模式下，用户主动开启后，麦克风持续发送声音，不需要按键。

规则：

```text
自由说话必须由用户主动开启。
自由说话不能作为首次进入房间的默认发送状态。
即使本地保存的语音模式偏好是自由说话，下次进入房间也不能自动开麦。
如果用户上次偏好自由说话，下次进入房间时可提示“上次使用自由说话，点击开启”。
```

切换到自由说话时，给轻提示：

```text
自由说话已开启，朋友可以听到你的麦克风声音。
```

验收：用户可以切换到自由说话；切换后持续发送；切回按键说话后停止持续发送；下次进入房间不会因上次偏好自由说话而自动开麦。

### 7.5 自己静音 / 取消静音

静音是覆盖状态，不改变底层语音模式。

发送声音必须同时满足：

```text
未静音
已连接
设备可用
当前语音模式的发送条件成立
```

具体行为：

```text
按键说话模式下静音：即使按住快捷键也不发送。
自由说话模式下静音：停止持续发送，但底层模式仍是自由说话。
静音期间可以切换语音模式。
取消静音后，按当前语音模式规则工作。
```

MVP 不支持本地静音某个成员、单独调节成员音量或房主远程静音成员。

验收：静音/取消静音入口清楚；静音状态明显可见；静音不会因切换窗口丢失；静音优先级高于按键说话和自由说话。

### 7.6 麦克风设备选择与测试

设备规则：

```text
默认麦克风设备 = 跟随系统默认麦克风。
用户可以指定某个麦克风设备。
用户选择后立即用于语音发送。
应用重启后保留选择。
如果指定麦克风不可用，自动回退到系统默认麦克风。
如果系统默认麦克风也不可用，显示“未检测到可用麦克风”。
```

测试麦克风只测试输入音量，不做本地回放。

设置页提供：当前麦克风设备、麦克风设备列表、输入音量条、无输入提示。

权限或设备不可用：

```text
麦克风权限失败：提示“无法使用麦克风，请检查系统权限”，提供“打开设置”和“仅收听进入”。
麦克风不可用：用户仍可进入房间，以“仅收听 / 已静音”状态进入，不发送声音。
麦克风恢复后，不自动开麦，仍需用户手动取消静音或开启语音。
```

验收：用户可以看到和切换麦克风设备；切换后立即用于语音发送；输入音量条能反映当前输入；设备不可用时可仅收听进入房间。

### 7.7 输出设备选择与整体输出音量

输出设备规则：

```text
默认输出设备 = 跟随系统默认输出设备。
用户可以指定某个输出设备。
用户选择后立即用于房间语音播放。
应用重启后保留选择。
如果指定输出设备不可用，自动回退到系统默认输出设备。
```

输出设备不可用时：

```text
优先回退系统默认输出设备。
如果仍不可用，允许进入房间。
房间页明显提示：输出设备不可用，可能无法听到语音。
提供“打开设置”和“继续进入房间”操作。
不影响用户发送语音，前提是麦克风可用且未静音。
```

整体房间语音音量只控制用户听到的房间成员语音总音量；不控制 Windows 系统音量、游戏音量、提示音，也不支持按成员单独调节音量。

验收：用户可以看到、切换并保存输出设备；整体输出音量调整后立即生效并保存。

### 7.8 连接状态与重连

状态类型：

```text
连接中
已连接
重连中
连接失败
已断开
```

首次进入房间：

```text
输入有效邀请码后进入房间页。
房间页显示语音连接状态：连接中。
连接成功后显示：已连接。
连接失败后显示可操作错误：重试 / 离开房间。
进入房间连接超时为 15 秒。
```

自动重连：

```text
语音连接异常后自动进入“重连中”。
自动重连最多持续 30 秒。
重连中必须停止发送语音，不能让用户误以为朋友还能听见。
30 秒内重连成功则恢复原成员状态。
30 秒内失败则进入“连接失败”，用户可以手动重试或离开房间。
```

其他成员视角：断线成员暂时保留在原位置，显示重连中 / 连接异常；30 秒失败后显示短提示并从在线成员列表移除。

弱网边界：

```text
MVP 不针对弱网提供专门降级策略。
弱网、断网、明显网络拥塞、异常 VPN 不属于正常网络验收范围。
网络异常时，产品只要求做到：明确连接状态、自动重连、失败后提供重试 / 离开房间。
MVP 不承诺在弱网下保持连续语音。
MVP 不承诺自动降低音质或切换传输策略。
```

验收：连接异常时用户能看到明确状态；重连中不发送语音；重连失败有重试和离开操作；其他成员能看到短暂连接异常状态。

### 7.9 系统托盘与轻量提示

系统托盘用于关闭主窗口后保持应用和语音连接运行。

托盘菜单至少包含：

```text
显示主窗口
离开房间
退出 echo
```

轻量提示规则：

```text
只做应用内视觉提示。
不播放提示音。
不弹 Windows 桌面通知。
不做系统级弹窗。
```

提示场景：

```text
成员加入
成员离开
连接异常
重连成功
复制邀请码成功 / 失败
进入系统托盘
设备切换成功 / 失败
```

验收：关闭窗口不离开房间且不中断语音；托盘入口能恢复主窗口、离开房间和退出应用；关键状态变化有视觉提示但不打扰游戏。

### 7.10 设置页、关于弹窗与反馈入口

设置页包含：昵称、随机头像、语音模式偏好、按键说话快捷键、麦克风设备、输出设备、输入音量条、整体房间语音音量、关于弹窗入口、反馈入口、隐私说明入口或简短说明。

关于弹窗包含：

```text
产品名称
当前版本号
当前阶段：朋友内测版
腾讯问卷反馈入口
轻量隐私说明
GitHub Releases 新版本下载渠道说明
```

外部反馈入口：

```text
点击“反馈问题”后，使用系统默认浏览器打开腾讯问卷外部表单。
发布朋友内测包前，反馈入口必须配置为有效可访问的腾讯问卷地址。
```

外部反馈表单建议收集：问题类型、问题描述、发生场景、联系方式（可选）、应用版本、Windows 版本（用户可手填）。

MVP 不做应用内反馈表、反馈邮箱作为主反馈方式、内测群作为主反馈方式、自动上传日志、附带诊断信息、工单系统或截图上传。

## 8. 体验标准

### 8.1 轻量体验

MVP 使用体验型轻量标准，不在 PRD 中写死 CPU、内存或安装包体积指标。

用户可感知目标：

```text
应用启动后应快速进入可操作状态。
房间语音期间不应明显影响游戏帧率、输入响应或系统切换。
托盘常驻时不应让用户感知到持续打扰或明显资源占用。
设置、房间页、托盘入口应保持简单。
```

产品体验上应避免：复杂首页、广告、信息流、推荐内容、大型社区入口、过多弹窗、强制登录、强制绑定手机号、好友/频道/聊天等重功能入口。

### 8.2 快速路径

用户应能在很短路径内完成：创建房间、复制邀请码、加入房间、开始说话。

MVP 内测最低目标：两名新用户从打开应用到进入同一临时房间并完成双向语音，≤ 60 秒。

### 8.3 稳定和状态清楚

语音状态必须回答用户的核心疑问：

```text
我现在到底有没有开麦？
朋友能不能听见我？
我是不是断线了？
按键说话有没有生效？
麦克风有没有输入？
我能不能听到朋友？
```

### 8.4 可接受语音质量

MVP 使用体验型语音质量标准，不在 PRD 中写死具体网络延迟、码率或丢包技术指标。

正常网络：用户所在网络能稳定访问互联网；不包含主动断网、弱网模拟、大文件上传下载导致的明显拥塞、异常 VPN、系统级网络限制或其他已知网络异常。

可接受语音质量：

```text
2-10 人房间内，成员能听清连续句子含义。
报点、提醒、战术指令不会因为语音延迟而失去沟通价值。
不应长期出现明显爆音、严重断续、持续电流声。
偶发卡顿允许，但必须可恢复，且状态提示清楚。
多人同时说话时允许声音重叠，但不应导致应用卡死或全部无声。
```

## 9. 隐私、安全与日志边界

### 9.1 隐私原则

```text
不要求注册账号。
不要求手机号。
不提供好友关系。
昵称、随机头像和设备偏好保存在本地。
房间语音仅用于实时通信。
MVP 不录制、不保存、不回放语音。
MVP 不自动上传诊断信息。
```

朋友内测阶段不做强制协议勾选；公开发布前重新评估是否需要正式用户协议和隐私政策。

### 9.2 关于弹窗隐私文案

```text
echo 无需注册账号或手机号。
昵称、随机头像和设备偏好仅保存在本机。
房间语音仅用于实时通信。
echo 不录制、不保存、不回放语音。
echo 不会自动上传诊断信息。
反馈问题时，请勿填写敏感个人信息。
```

该说明用于朋友内测阶段，不等同于正式隐私政策。

### 9.3 治理边界

MVP 不支持举报、封禁、黑名单、踢出成员、撤销邀请码、房间密码。

如果邀请码泄露或陌生人进入，用户可以离开当前房间，重新创建新的临时房间，并分享新的邀请码给朋友。

### 9.4 日志边界

MVP 可以保留详细客户端本机日志和服务端本机日志，用于开发和内测排障。但日志不会自动上传，腾讯问卷不会自动附带日志。

日志不得记录语音内容、音频数据、长期明文邀请码历史、连接凭证明文或敏感请求体。排障时可由用户或开发者手动导出、截图或复制错误信息。

## 10. 内测验收与指标

### 10.1 内测规模与分发

```text
v0.1 朋友内测建议控制在 20-50 人。
内测采用邀请制，不开放公开下载入口。
若内测规模预计超过 100 人，需要先重新评估自动更新、反馈处理、治理与运维能力。
```

分发规则：

```text
v0.1 Windows x64 安装包通过项目发布流程产出。
内测安装包通过 GitHub Releases 提供下载。
内测用户通过指定 GitHub Releases 下载地址获取安装包。
应用内不做自动更新。
发现阻断问题时，通过原分发渠道通知用户下载新版。
旧版本不在应用内强制升级。
关于弹窗展示当前版本号和 GitHub Releases 新版本下载渠道说明。
```

### 10.2 必过验收场景

| 场景 | 必过标准 |
| --- | --- |
| 两人快速开黑 | 两名用户无需注册，通过邀请码进入同一房间并双向语音 |
| 3-10 人房间 | 3-10 人正常语音；第 11 人加入失败并看到明确提示 |
| 后台游戏按键说话 | 应用后台或托盘中，游戏前台时，按下发送、松开停止 |
| 设备选择与排障 | 麦克风/输出设备可切换；输入音量条有效；设备不可用时提示清楚 |
| 断线重连 | 30 秒内恢复；超过 30 秒进入连接失败并提供操作 |
| 托盘行为 | 点击窗口 X 不离开房间；托盘可显示主窗口、离开房间、退出应用 |
| 隐私与低打扰 | 用户能看到隐私说明；不提示音；不弹 Windows 桌面通知；不自动上传诊断信息 |

游戏场景覆盖类型：

```text
前台独占 / 全屏游戏
无边框窗口游戏
管理员权限运行的游戏
带反作弊或限制全局快捷键的游戏
普通桌面场景，用于排除非游戏问题
```

PRD 只定义需要覆盖的游戏场景类型；具体游戏名称、测试步骤和通过记录放入独立验收清单或 `implement.md`。

### 10.3 MVP 最低验收阈值

| 指标 | MVP 目标 |
| --- | --- |
| 冷启动到首页可操作 | 已安装后打开应用，到首页可操作目标 ≤ 5 秒 |
| 两名用户完成首次开黑 | 两名新用户从打开应用到进入同一临时房间并完成双向语音，≤ 60 秒 |
| 10 人房间连续语音 | 10 人房间至少连续语音 20 分钟，不出现全员掉线或全员无声 |
| 正常网络下语音沟通 | 说话内容可听清，主观延迟不明显影响游戏沟通 |
| 后台按键说话 | 游戏在前台、应用在后台或托盘时，连续 10 次按下 / 松开快捷键无明显漏触发 |
| 核心流程独立完成 | 内测用户无需他人远程指导，可完成创建房间、复制邀请码、加入房间、说话、切换设备和退出 |
| 人数上限提示 | 第 11 名用户加入失败时，看到明确原因和可操作提示 |

### 10.4 观察指标

| 指标 | 观察目标 |
| --- | --- |
| 创建房间成功率 | 观察失败原因，优先修复阻断进入房间的问题 |
| 邀请码加入成功率 | 观察无效码、过期码、满员、连接失败等失败分布 |
| 首次听到别人声音比例 | 观察麦克风、输出设备、权限和网络问题 |
| 首次成功说话比例 | 观察按键说话、静音、麦克风不可用和权限问题 |
| 单次语音时长 | 观察是否能覆盖一局游戏 |
| 中途掉线与重连 | 观察断线次数、重连是否成功、用户是否理解状态 |
| 因问题退出比例 | 观察用户退出前遇到的阻断问题 |
| 用户是否愿意下次继续用 | 观察轻量、低打扰、语音稳定是否成立 |

## 13. 风险与产品应对

| 风险 | 表现 | MVP 应对 |
| --- | --- | --- |
| 用户不知道自己是否开麦 | 误以为朋友能听到或听不到 | 房间页明确展示语音状态、按键状态和输入反馈 |
| 按键说话在游戏中不生效 | 核心体验失败 | 后台按键作为 P0；设置页提供兼容性提示；房间页显示按键状态 |
| 麦克风设备选错 | 没声音 | 提供麦克风设备选择、输入音量条、测试麦克风 |
| 输出设备选错 | 听不到朋友 | 提供输出设备选择和整体房间语音音量 |
| 关闭窗口后语音中断 | 用户以为只是关窗口 | 关闭窗口固定进入系统托盘 |
| 房间被陌生人加入 | 邀请码外泄 | MVP 通过重新创建临时房间规避；后续支持撤销邀请码和房主踢人 |
| 功能越做越重 | 违背产品初衷 | 坚持 MVP 不做账号、社区、聊天、固定房间等 |
| 隐私担忧 | 用户担心录音或上传 | 明确不录音、不保存语音、不自动上传诊断信息 |
