# 本地匿名身份、昵称、随机头像和本地设置

## 目标

让 Windows 客户端在无账号条件下，为当前电脑建立可恢复的匿名身份与个人偏好。首次启动必须生成并持久化可用身份；后续启动必须恢复已保存的昵称、随机头像和设备/语音偏好；配置文件损坏不得阻止应用继续以安全默认值启动。

## 已确认事实与约束

- Issue #17 是本任务的需求来源；#6 已关闭，桌面与前端测试骨架存在。
- `anonymous_id` 是本机匿名身份，只用于临时房间成员识别、重连和房主标识；不是账号、登录凭证、好友身份或跨设备资料（`CONTEXT.md:27-29`，`docs/adr/0012-client-generated-anonymous-identity.md:3`）。
- 本地设置由 Wails/Go 层保存在当前 Windows 用户的应用配置目录；业务服务不保存个人偏好（`design.md:166-187`，`docs/adr/0017-local-settings-in-user-config.md:3`）。
- 默认按键说话快捷键为 `V`、默认语音模式为 `push_to_talk`、默认整体输出音量为 `100`（`implement.md:1744-1765`）。
- 保存自由说话偏好绝不等于在新房间自动发送；每次进入房间都需要本房间内的明确开启动作（`prd.md:233`，`prd.md:433-441`）。
- 本任务不接管尚未实现的首次昵称页或设置抽屉 UI；它提供这些后续界面调用的持久化与语音状态合同。

## 需求

### R1：本机设置存储

Go 配置模块必须读取和写入当前 Windows 用户应用配置目录中的 JSON 设置。它必须拥有文件路径、默认值、序列化、身份/头像生成和损坏数据回退；调用方不得重复实现这些规则。

### R2：身份和头像生命周期

首次加载无设置时必须生成非空 `anonymous_id` 和随机 `avatar_id`，并持久化后再返回。重新随机头像必须生成新的非空 `avatar_id`、保留其他已保存设置并持久化。删除设置后首次加载的行为与无设置相同。

### R3：保存字段与默认值

必须持久化并恢复：匿名身份、昵称、随机头像、按键说话快捷键、麦克风设备、输出设备、语音模式偏好和整体输出音量。设备标识为空表示跟随系统默认设备。昵称允许重复，存储层不引入账号或唯一性语义。

### R4：安全恢复

缺失、截断、非 JSON 或含无效安全关键字段的配置，必须归一化为可用设置：匿名身份和头像非空，语音模式回退到按键说话，默认快捷键与音量有效。恢复不得 panic 或使启动失败。

### R5：语音发送状态

前端必须拥有明确、可测试的语音发送判定：连接、可用麦克风和未静音是必要条件；按键说话还要求按住快捷键；自由说话还要求本房间显式开启。已保存的 `free_talk` 偏好不能单独使新房间发送音频。

### R6：前端设置合同

前端必须提供与 Go 结构字段一一对应的强类型 `LocalSettings` 及安全默认值，供首次昵称页、设置抽屉和后续 Wails bridge 使用。该合同不保存房间历史、自动加入、关闭窗口偏好、账号数据或 LiveKit token。

## 验收标准

- [ ] AC1：空配置路径首次加载时生成并写入非空匿名身份和随机头像；默认快捷键为 `V`、语音模式为按键说话、输出音量为 100。
- [ ] AC2：保存昵称后用新 Store 实例再次加载，能恢复昵称和原匿名身份。
- [ ] AC3：重新随机头像后，头像值改变、非空，且再次加载保持新值；其他设置不丢失。
- [ ] AC4：麦克风、输出设备、整体输出音量、快捷键和语音模式偏好都能跨 Store 实例恢复。
- [ ] AC5：自由说话偏好可持久化；新房间的 `freeTalkEnabledInRoom=false` 时语音发送判定为 false；显式开启后才允许发送。
- [ ] AC6：损坏 JSON 配置加载不 panic，返回并持久化可用回退设置，且不自动发送音频。
- [ ] AC7：`go test ./internal/config -v` 和 `npm run test:run -- src/settings/settings.test.ts src/state/voiceState.test.ts` 均通过。

## 非目标

- 账号、登录、云同步、跨设备恢复、好友关系或服务端个人资料。
- 房间历史、自动加入上次临时房间、关闭窗口行为偏好或 LiveKit token 持久化。
- 头像上传、头像库、裁剪或编辑。
- 首次昵称页、设置抽屉、设备枚举/实际切换、媒体发送、托盘行为或房间 UI 的实现。

## 规划结论

产品、架构和 UI 文档已回答范围与安全语义；没有阻塞规划的产品问题。用户已明确要求完成实施与独立检查，因此本 PRD 可进入设计审查与激活。
