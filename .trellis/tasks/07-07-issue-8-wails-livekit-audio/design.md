# Design：验证 Wails 3 + WebView2 + LiveKit 音频路径

## First-principles analysis

### 1. Challenge assumptions

- 未验证假设：Wails 3 WebView2 可以稳定执行 LiveKit JS 的 WebRTC 连接、麦克风权限、音频发布和远端播放。
- 未验证假设：LiveKit JS 在 WebView2 中的音频播放不会因为自动播放策略、权限、WebView 生命周期或托盘隐藏而失败。
- 潜在错误类比：浏览器 Chrome/Edge 能跑 LiveKit，不等于 Wails WebView2 宿主内一定可用。
- 潜在过度实现：为了验证媒体路径而实现临时房间、邀请码、业务 token 服务、正式房间 UI 或 Go 音频管线。

### 2. Bedrock truths

- echo MVP 的实时语音必须让用户“能说话、能听见”；如果媒体路径不可用，临时房间业务逻辑无法弥补。
- WebRTC 音频在客户端侧需要四个最小事实同时成立：连接 LiveKit、获得麦克风 track、发布本地音频、订阅并播放远端音频。
- Wails Go shell 的 fundamental duty 是窗口/托盘/native 能力；音频媒体在项目 ADR 中被限定在 WebView2 + LiveKit JS。
- LiveKit join token 是短期敏感凭据；验证可以手工粘贴，但不能提交或写入长期文档。
- HITL 是必要条件，因为 Windows WebView2、真实麦克风、音频输出和远端播放不是纯单元测试能证明的。

### 3. Rebuild from truths

从上述事实出发，本任务只需要一个 spike 页面：输入 LiveKit URL 和 token，创建 `Room`，监听连接/订阅/断开事件，调用 `room.connect(url, token)`，调用 `room.localParticipant.setMicrophoneEnabled(true)`，对远端 audio track 执行 `track.attach()` 并挂载到 DOM，最后支持 `room.disconnect(true)` 或等价清理。任何业务房间、邀请码、服务端 token 签发、正式 UI 都不是证明媒体路径的必要条件。

### 4. Contrast with convention

常规思路可能会直接搭一个小型产品房间流来“顺便”验证媒体，但这会把业务服务、token、成员状态和 UI 复杂度混入音频路径判断。这里的本质问题只有媒体路径是否成立，因此先隔离 LiveKit/Wails/WebView2 音频，降低结论噪声。

### 5. Conclusion

本任务的最小正确设计是一个隔离的 LiveKit 音频 spike，加上 Windows 双客户端 HITL 记录。如果该 spike 失败，应暂停正式媒体实现，而不是在本任务内引入 Go 音频栈或 Electron fallback。

## Architecture boundaries

### In scope

- `apps/desktop/frontend/package.json`：添加 `livekit-client` 依赖。
- `apps/desktop/frontend/src/spike/LiveKitAudioSpike.tsx`：新增音频 spike 页面。
- `apps/desktop/frontend/src/App.tsx`：临时渲染音频 spike 页面。
- `docs/spikes/wails-livekit-audio.md`：记录自动验证、Windows HITL 结果、失败/限制和后续约束。

### Out of scope

- `services/api` 不参与本 spike。
- 不新增正式 `frontend/src/livekit/livekitClient.ts`。
- 不新增业务 token 签发、OpenAPI、WebSocket、临时房间或成员列表。
- 不修改 Go 音频能力；Go 层只提供现有 Wails shell 与托盘行为。

## Runtime flow

```text
Tester opens Wails dev app
  -> LiveKitAudioSpike renders URL/token inputs
  -> Tester pastes public LiveKit WSS URL and short-lived token
  -> Click connect
  -> Room connects to public LiveKit
  -> WebView2 requests microphone permission if needed
  -> localParticipant.setMicrophoneEnabled(true) publishes local audio
  -> second client joins same LiveKit room through an official/self-hosted test page or another machine
  -> RoomEvent.TrackSubscribed receives remote audio
  -> track.attach() creates audio element and appends it to DOM
  -> each side speaks and confirms the other side hears audio
  -> tester clicks disconnect or exits app
```

## Component design

`LiveKitAudioSpike` state:

- `url`: LiveKit WebSocket URL.
- `token`: short-lived token, stored only in React state for the session.
- `status`: `idle | connecting | connected | failed | disconnected`.
- `error`: sanitized error string from thrown error.
- `events`: short non-secret status log for UI visibility.
- `roomRef`: current `Room | null`.
- `remoteElementsRef`: attached audio elements for cleanup.

Important behaviors:

- Disable connect while already connecting/connected.
- On connect, disconnect existing room first to avoid duplicate microphone capture.
- Set event handlers before `connect` so connection and track events are visible.
- On `TrackSubscribed`, only attach `Track.Kind.Audio`; ignore video.
- On `TrackUnsubscribed` or disconnect, detach tracks/remove appended audio elements.
- On unmount, disconnect and clean attached audio elements.
- Use `setMicrophoneEnabled(true)`, not camera APIs.

## HITL environment decision

- Use a public LiveKit WSS service for manual validation.
- Use a second client outside the Wails spike when possible, such as a LiveKit official/self-hosted test page or another machine.
- Do not record the token, API key, API secret, or reusable room credential in repository files.
- Treat restrictive NAT/firewall failures as deployment/network evidence, not immediate proof that the WebView2 media path is invalid.

## Compatibility notes

- Use `livekit-client` directly rather than `@livekit/components-react`; the spike does not need React LiveKit components.
- Keep styles local/simple, consistent with existing spike pages.
- WebView2 audio autoplay can require user interaction. The connect button provides a user gesture before remote audio playback.
- The existing tray close-to-hide behavior from Issue #7 should preserve the WebView. This task records whether audio remains usable in the tested flow, but does not reimplement tray.

## Security and privacy notes

- Token is entered manually and held in memory only.
- Do not log token or write it to docs.
- Documentation may record token source as a category, for example “generated by LiveKit CLI/dev helper”, but not the token value.
- Do not record voice content or audio data.

## Rollback

- Remove `LiveKitAudioSpike.tsx`.
- Restore `App.tsx` to its previous spike route if needed.
- Remove LiveKit dependency if no longer used.
- Remove or mark `docs/spikes/wails-livekit-audio.md` as failed if validation blocks progress.
